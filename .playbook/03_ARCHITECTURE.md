# Architecture

## Current architecture (as of Milestone 14)

```
index.html            Thin shell: markup skeleton, <link> to assets/css/site.css, script tags in load
                       order, and a single `App.boot();` call. No inline logic (Milestone 13 — see
                       12_DECISIONS.md DR-019) and no Tailwind CDN (DR-015 — presentation is
                       assets/css/site.css design tokens instead). Product-mode content between
                       `<!-- PRERENDER:START/END:x -->` comment markers is static HTML baked from
                       data/*.json by `npm run prerender` (Milestone 14, DR-022) — see "Prerendering"
                       below.
assets/css/site.css    Design tokens (`data-theme` light/dark), layout, and every component's styling.
assets/js/icons.js     Icons namespace — inline SVG path data, no external icon font/CDN.
assets/js/render.js    Render namespace — pure markup builders (see 10_COMPONENT_LIBRARY.md). Also
                       require()'d directly by scripts/prerender.js (Node), not just the browser.
assets/js/filters.js   CompanyFilters namespace — company filter/sort/URL-state logic (Milestone 8/9),
                       the single source of truth for sort options and column-header sort targets.
assets/js/search.js    Search namespace — buildIndex, query, rank over data/*.json (Milestone 5+).
assets/js/ui.js        UI namespace — theme toggle, scroll-spy, custom selects, command palette,
                       nav drawer, debounce helper.
assets/js/app.js       App namespace — App.boot() fetches data/*.json in parallel and wires up
                       every interactive container (company pagination/filters/sort, learning-table
                       pagination, copy-link). Extracted from index.html's inline script (DR-019).
data/                  Runtime JSON only: chapters, companies, site, roadmaps, and six learning
                       reference files (glossary, technologies, career_paths, interviews, templates,
                       resources) — see 09_DATA_MODEL.md and 11_COMPANY_SCHEMA.md.
scripts/               Dev-only Node tooling: verify-*.js (require render.js/filters.js/search.js
                       directly, no browser needed), prerender.js / verify-prerender.js / lib/
                       prerender-core.js (Milestone 14, DR-022), ui-smoke-*.mjs (Playwright),
                       schema/seed helpers, git-hooks/pre-commit, install-git-hooks.js. Never loaded
                       by index.html.
sitemap.xml            Generated by scripts/prerender.js from data/site.json's `seo.siteUrl` — commit
                       like any other file, do not hand-edit (Milestone 14, DR-022).
robots.txt             Generated by scripts/prerender.js — same rule as sitemap.xml.
archive/               Historical company research, seed data, and legacy build scripts moved out of
                       the live path in Milestone 13 (DR-017) — reference only, not a second database.
.playbook/             Documentation and governance (this directory).
.github/               Repository automation (CI workflow) and AI-agent instructions.
```

**Note:** `index.html` must be served over HTTP for local viewing — `fetch()` against `file://` is blocked by
browser CORS policy. See `12_DECISIONS.md` DR-005 and `17_TESTING_GUIDE.md`.

## Script load order and cross-file references

`index.html` loads `icons.js`, `render.js`, `filters.js`, `search.js`, `ui.js`, then `app.js`, as classic
(non-module) `<script>` tags, followed by a one-line `App.boot();`. Each file is a single `const` object
(`Render`, `CompanyFilters`, `Search`, `UI`, `Icons`, `App`) attached to `window` at the bottom for clarity, guarded
by `if (typeof module !== 'undefined' && module.exports)` so the same file can also be `require()`'d directly by
Node (`scripts/verify-*.js`) without a browser or bundler.

Classic `<script>` tags on one page share a single top-level lexical scope for `const`/`let`, so a later-loaded
file's namespace (e.g. `CompanyFilters`, declared in `filters.js`, loaded after `render.js`) is still resolvable as
a bare identifier from an earlier file's function body (`render.js`'s `companyFilterActive`) — because that
function only actually *runs* after every script tag has finished loading and executing, once `App.boot()`'s
`fetch()` promises resolve. This is why `render.js` can reference `CompanyFilters` even though it loads first; see
the `Render._companyFilters()` comment in `assets/js/render.js` for the one place this needed an explicit Node
fallback (Node's isolated `require()` doesn't share that browser-only scoping trick).

## Prerendering (Milestone 14, DR-022)

**Problem:** `index.html`'s `<body>` ships with every content container empty; real markup only exists after
`assets/js/app.js`'s `App.boot()` runs. Search engines that don't execute JavaScript (and many non-Google
crawlers/link-preview bots that don't) see an empty page.

**Approach — bake, don't hydrate:** `scripts/prerender.js` (shared logic in `scripts/lib/prerender-core.js`)
`require()`s `assets/js/render.js` directly in Node — the same pattern `scripts/verify-render.js` already uses —
and calls the exact same `Render.*` functions with the exact same arguments `App.boot()` calls in the browser, for
the always-published **product-mode** view only (`?mode=dev` is never prerendered). The resulting HTML strings are
spliced into `index.html` between fixed `<!-- PRERENDER:START:x --><!-- PRERENDER:END:x -->` comment-marker pairs
(one pair per container: page header, both search shells, disclaimer, sidebar label, table of contents, `<main>`,
footer, plus a `<link rel="canonical">` and a JSON-LD `<script>` in `<head>`). `assets/js/app.js` is **not**
hydration-aware and requires no changes — it still fully re-renders every container on load, exactly as before
Milestone 14. Because both the baked markup and the client re-render come from the identical `Render.*` functions
and the identical data, the re-render produces byte-for-byte the same output in the common case, so there is no
visible "flash of different content" — just a redundant, imperceptibly fast re-render. This trades a small,
constant amount of wasted client-side work for zero hydration-mismatch bug surface (a real and easy-to-get-wrong
class of bugs in true hydration systems), which was judged the right trade-off for this codebase's size and audit
history (see `12_DECISIONS.md` DR-022 for the alternatives considered).

**Staleness enforcement:** `scripts/verify-prerender.js` reruns the same generation function in memory and
byte-compares it against `index.html`, `sitemap.xml`, and `robots.txt` on disk, failing loudly if they don't match
— i.e. someone edited `data/*.json` without re-running `npm run prerender`. It is the last step of `npm run verify`,
so it runs locally (pre-commit hook), and in CI (`Verify scripts` job) — the same command, the same one enforcement
path for both.

**`sitemap.xml` / `robots.txt`:** also generated by `scripts/prerender.js` from `data/site.json`'s `seo.siteUrl` —
treat them as generated files (like `index.html`'s baked sections), not hand-edited ones.

## Data flow

1. `index.html` loads all six `assets/js/*.js` files, then calls `App.boot()`.
2. `App.boot()` `fetch()`es every `data/*.json` file in parallel (`Promise.all`), then calls the matching `Render.*`
   method per section; returned markup is injected into container elements.
3. Render functions do not own or mutate the underlying data — they take data + options and return an HTML string.
4. `App.boot()` also builds the search index (`Search.buildIndex`) and wires every interactive container
   (`App.wireCompanyPagination`, `App.wirePaginatedContainer`, `UI.wireSiteSearch`, `UI.wireCommandPalette`, ...).
5. Company-table filter/sort state and the site-search facet state are the same shared object (see DR-019) —
   mutating it from either the filter toolbar, a table column-header click, or a search facet chip keeps both in
   sync without manual field-by-field copying.

## Constraints that shape every architectural decision

- No build step at request/deploy time — static files served as-is (GitHub Pages compatible). The one narrow,
  documented exception is `npm run prerender` (Milestone 14, DR-022), a pre-commit step whose output is committed
  like any other file.
- No frameworks — vanilla JavaScript only; styling is `assets/css/site.css` (design tokens + utility-like classes),
  not a CSS framework or CDN.
- GitHub Pages, offline, responsive, accessible, and print-friendly at all times.
- Renderer reads data; UI never owns data (data and presentation stay separable at every layer).
- ESLint + Prettier (Milestone 13, DR-021) run in CI and a local pre-commit hook, but are dev-time-only tooling —
  they add no runtime dependency and change nothing that ships to the browser.

## Why a single HTML shell is acceptable (for now)

An `index.html` shell plus a small set of namespaced JS modules plus JSON data files maximizes portability (works
offline when served, works unmodified on GitHub Pages, easy to fork) at the cost of some scalability headroom. This
is an accepted trade-off while content volume is small (`data/companies.json` is ~119 records / ~222KB as of
Milestone 13 — see `12_DECISIONS.md` DR-020 for the concrete threshold and future chunking strategy). If the data
model or content volume genuinely outgrows this layout, that change goes through `12_DECISIONS.md` before being
implemented.
